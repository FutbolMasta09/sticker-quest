## Program Sync (Short-Term And Long-Term)

This section keeps all active plans aligned so day-to-day execution supports long-term goals.

### Active Program Order

- **Track 1 (now):** Sticker Quest delivery against roadmap phases (Phase 1 -> 4 first).
- **Track 2 (parallel planning):** Anchor's Desk design prep and low-risk audience seeding.
- **Track 3 (long-term brand):** Inkborne universe canon/IP development and community growth.

### Sync Rules Between Plans

- Sticker Quest is the primary delivery commitment until its milestone gates are complete.
- Anchor's Desk build starts from its timeline trigger (post Sticker Quest launch window), while design prep continues in small blocks.
- Inkborne community posts continue weekly, but must stay inside time cap and never delay roadmap gates.

### Milestone Hand-Off Map

- **Sticker Quest Phase 1 complete:** keep Anchor's Desk in design mode only, no full production start.

## Weekly Operating Rhythm (Day-By-Day Structure)

Use this as the default weekly structure. Adjust hours by energy, but keep day themes consistent.

### Default Day Themes

- **Monday - Deep Build:** primary coding work on current Sticker Quest gate tasks.
- **Tuesday - Deep Build:** continue implementation + focused bug fixing/testing prep.
- **Wednesday - Design/Planning:** Anchor's Desk design notes, Inkborne canon planning, architecture decisions, backlog shaping.
- **Thursday - Deep Build:** ship priority tasks, close loop on in-progress work.
- **Friday - Community + Business:** posts, replies, KPI update, monetization/admin tasks.
- **Saturday - Weekly Program Review + Next Week Planning:** 30-minute steering review, next-week schedule lock, then optional flex block (catch-up if behind, advance if ahead, or rest).
- **Sunday - Rest/Church/Light Catch-up (optional):** recovery day or minimal backlog cleanup only.

### Saturday Family Priority Rule

- Saturday planning block is fixed at 30 minutes.
- Any Saturday work after planning is optional, not mandatory.
- If behind: use optional block for catch-up on must-ship work.
- If on track or ahead: use optional block to work ahead only if energy/time are available.
- Family time is the default priority; skip optional block when personal/family time needs it.

### Daily Block Recipe (Use For Any Workday)

- Block 1: Primary objective (highest-value ship task)
- Block 2: Secondary objective (supporting task)
- Block 3: Comms/admin/community (time-capped)
- 10-minute closeout log:
  - What was planned
  - What got done
  - What slipped and why
  - First task for tomorrow

### If Energy Is Low

- Keep the day theme, reduce scope:
  - Build day -> one bug or one small feature slice
  - Planning day -> one design decision + one backlog update
  - Community day -> one post + two meaningful replies

## Community And Patreon Strategy

- Build community intentionally, not passively.
- Keep it lightweight so it supports shipping instead of stealing time.
- Use a dual-brand lane:
  - **Public Brand Lane (primary):** Inkborne universe and future game development.
  - **Builder Lane (secondary):** Sticker Quest progress, lessons learned, and reliability wins.
- Use a two-step support model:
  - Step 1: `Buy Me a Coffee`/`Ko-fi` as a low-friction tip layer.
  - Step 2: Patreon only when you can reliably deliver recurring member value.

### Dual-Brand Content Split

- Default split: `80%` Inkborne/game-universe content, `20%` Sticker Quest builder updates.
- Keep Sticker Quest updates process-focused (what you learned, what you fixed, what shipped), not broad consumer marketing.
- Use one consistent creator identity across both lanes so audience overlap compounds.

### Community Build System (While Developing)

- **Cadence (weekly):**
  - 1 build update post (what moved forward this week)
  - 1 world-building or design post (Inkborne lore, mechanics, art direction)
  - 3-5 lightweight replies/comments to relevant creators or potential players
- **Pillars (repeatable topics):**
  - Build log (before/after progress)
  - Design decisions (why this mechanic, why this style)
  - Problem-solving stories (bug -> fix -> lesson)
  - World snippets (character, faction, setting)
- **Engagement loop:**
  - End at least one post each week with a clear question.
  - Save responses in a short notes file.
  - Use the top recurring response as next week's content seed.
- **Time cap:**
  - Community work max `3-4 hours/week` until core roadmap gates are stable.

### Launch Triggers

- **Tip jar trigger (start early):**
  - At least 1 consistent audience channel is active.
  - You can post updates at least once weekly.
- **Patreon trigger (start later):**
  - At least 20-30 people explicitly express support intent.
  - You can ship 2+ member updates monthly for 3 months in a row.
  - Core product roadmap deadlines are still being met.

### Simple Member Offer (When Patreon Starts)

- `$3`: supporter badge + monthly dev update
- `$7`: behind-the-scenes + roadmap poll vote
- `$15`: monthly supporter Q&A or build recap

### 30-Day Community Kickoff

- Week 1: choose one primary platform, publish creator bio and pinned post, publish first Inkborne build update.
- Week 2: publish one Sticker Quest builder update and one Inkborne world post, track support-intent signals.
- Week 3: repeat cadence and test one post format change (short video, carousel, or thread).
- Week 4: review which topic earned best engagement and lock next month's topic mix.

## Inkborne IP Protection Framework

- Keep a dated **idea provenance log** for universe canon, mechanics, names, factions, and story arcs.
- Separate your content into two buckets:
  - **Public-safe:** high-level themes, progress updates, dev process, non-critical snippets.
  - **Private-core:** key lore twists, signature mechanics, plot anchors, naming systems, and release plans.
- Before posting concept details, run a quick rule: "Would publishing this reduce uniqueness if copied?" If yes, keep it private.
- Put clear terms on public spaces (site/about/pinned post): no reuse of original lore/art/text without permission.
- Maintain monthly evidence backups (timestamps, drafts, files, screenshots) to support ownership history.
- For higher-stakes legal setup later, schedule an IP attorney review when budget allows.

## Human-First AI Collaboration System

- AI role: planning, outlining, research summaries, checklists, iteration options, editing passes.
- Your role: final creative choices, voice, canon decisions, character intent, and release decisions.
- Draft pipeline:
  - Step 1: You create rough concept notes.
  - Step 2: AI helps structure outlines and options.
  - Step 3: You write first creative draft in your own words.
  - Step 4: AI gives revision suggestions (clarity/pacing/consistency), not final replacement text.
  - Step 5: You approve and finalize.
- Attribution/ethics guardrail:
  - Do not publish raw AI output as final canon text.
  - Use AI-generated text as scaffolding, then rewrite in your own style.
- Quality gate before publishing:
  - Voice consistency with Inkborne canon
  - Originality check against your prior universe docs
  - Final human edit pass

## "Steal Risk" And "Accidental Copy" Risk Controls

- **Accidental copy protection:** when inspired by external creators, log source references and force a 3-part transformation (mechanic, framing, and tone) before using.
- **Steal risk reduction:** publish enough to build audience trust, but keep high-leverage canon/mechanics private until execution is underway.
- **Dispute readiness:** keep creation timeline records centralized and easy to export.
- **Platform behavior:** avoid posting complete design docs or full plot structures publicly.

## Quit/No-Quit Scorecard (Monthly)

Use this at the end of every month. If any red condition appears, do not quit yet.

### Green (Ready To Quit Soon)

- Runway cash is at least `9 months` (`$36k+`)
- Last 3 months average net income is `>= $5,000/month`
- No month in last 3 dipped below `$4,000`
- Product delivery rhythm is stable (planned work mostly ships)
- Acquisition is not from one fragile source only

### Yellow (Close, But Hold)

- Runway is `6-8 months` (`$24k-$35k`)
- 1-2 months at/above `$5,000`, but not 3 in a row
- Revenue is concentrated in one channel or one launch spike
- Retention/return behavior is unclear

### Red (Do Not Quit)

- Runway is below `6 months` (`<$24k`)
- Last 3 months average net income is `< $4,000`
- Revenue highly volatile with no repeatable pattern
- Deadlines repeatedly missed due to overload/scope drift

## Hard Decision Rule

- **Quit only when all are true:**
  - Runway `>= 9 months`
  - 3 consecutive months `>= $5,000 net`
  - 3-month average `>= $5,000`
  - No single channel contributes more than `60%` of new customer flow

## Accountability Thresholds And Interventions

Use these hard thresholds to protect the full-time solo-dev target.

### Target Capacity Band

- Baseline expectation: `20+` weekday hours (Mon-Fri combined).
- Preferred execution band: `20-28` weekday hours.

### Trigger Thresholds

- **Hours drift trigger:** weekly actual hours are `>20%` below plan for 2 consecutive weeks.
- **Finish-rate trigger:** finish rate below `60%` for 2 consecutive weeks.
- **Habit-risk trigger:** weekly habit risk score `>= 8`.
- **Scope trigger:** 3+ unplanned scope additions in one week.
- **Must-ship trigger:** must-ship item missed 2 weeks in a row.

### Required Interventions

- **If one trigger is hit:**
  - Reduce next-week planned workload by `20%`.
  - Add one protected buffer block.
  - Cap active tasks to 2 at a time.
- **If two triggers are hit in the same week:**
  - Freeze new scope for 7 days.
  - Execute only must-ship + blocker removal tasks.
  - Move hardest task to first work block on two weekdays.
- **If triggers persist for 4 weeks:**
  - Mark timeline status as `off track`.
  - Run monthly deep-audit recovery plan immediately.
  - Re-forecast full-time target date using current throughput.
