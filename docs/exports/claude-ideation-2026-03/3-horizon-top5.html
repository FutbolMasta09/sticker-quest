3-Horizon Expansion · Post-Market Top 5
Where each idea can go
H1 = build now, solo, low budget. H2 = after shipping H1, with expanded skills and moderate spend. H3 = moonshot, small team, better funding. Five moonshots total across the portfolio — one per idea, used deliberately.
Post-market #1 · Final score 4.79 · Tool
Prompt Forge
Tool Build Now No asset deps
Moonshot 1 of 5: From personal workbench to the default prompt testing layer for indie AI builders — a niche platform that compounds in value the longer you use it.
H1 — Build Now
Personal Prompt Workbench
1–2 weeks · solo · $0
Minimum lovable version
Prompt input → Claude API call → output display → manual rubric scoring (1–5 per criterion) → one-click save to localStorage → library list with tag filter and score sort. Nothing more.
Leave out
  • ✕Side-by-side variant comparison
  • ✕Prompt versioning or history
  • ✕AI-suggested rewrites
  • ✕Export, sharing, or any cloud sync
  • ✕Any analytics or score charts
Done when
You have saved 10 real prompts and loaded at least 3 of them back into the runner to improve them. That behavior is the product — not the code.
H2 — Scale Later
Prompt Lab
After H1 ships · moderate skills · low budget
3 upgrades
  • +Variant runner. Run 2–3 prompt variations simultaneously, score each, see diffs. The rubric finally earns its weight as a comparison tool.
  • +AI-suggested rewrite. After scoring, ask Claude to propose a rewrite targeting the lowest-scoring criteria. Meta-layer on top of the base loop.
  • +Supabase sync. Cross-device persistence. Only necessary when the library is large enough to be painful to lose — and you'll know when that is.
New risks
  • !Variant runner adds API cost fast — you're calling Claude 3x per session.
  • !AI rewrite suggestions may be obvious or circular — test before shipping as a feature.
Skills to gain first
Supabase auth + CRUD Multi-call state batching Diff UI rendering
H3 — Moonshot
Forge Platform
Small team · 6–12 months post-H1
Vision
A shared prompt library for indie AI builder teams — with public community rubrics, scored by real users, searchable by use case. Not enterprise. Not a marketplace. The GitHub for prompt iteration.
Roles needed
Product designer Backend / Supabase engineer Growth (community)
Budget assumption
$10K–$30K. Primarily API costs, hosting, and part-time design. No marketing budget needed if community seeding works — this audience distributes by word-of-mouth.
Why only later
The personal tool must prove the rubric mechanic works for one person before it earns the right to serve a community. Build the community on top of a behavior that already exists — not in hope of creating one.
Trigger: H1 → H2
You have used the tool personally for at least 2 weeks and found yourself wanting to test variants of the same prompt. Not before — H2 is not an upgrade to build speculatively.
Budget trigger: None — H2 is free. People trigger: Still solo.
Trigger: H2 → H3
At least 2 other people have asked to use the tool after seeing it, and you have evidence they would return after a first use — not just express interest once.
Budget trigger: $5K+ available. People trigger: 1 part-time collaborator committed.
De-scope under timeline pressure
Cut tag system first — tags are useful but the save + retrieve loop works without them. Cut library sort next. Ship: run → score → save → find.
Do not start H2 yet if
You have not personally used H1 for 10+ sessions. Upgrading a tool you have not validated is building on sand.
Market angle
The only personal prompt workbench built around a rubric you define — not a team logger, not a marketplace, not an analytics dashboard.
Post-market #2 · Final score 4.62 · Game · Tier 2
Time Loop Inbox
Game Build Now Edge 5/5
Moonshot 2 of 5: A puzzle anthology platform built around the loop-inbox mechanic — curated mystery episodes sold or subscribed to, with a creator mode for writers to build puzzles without code.
H1 — Build Now
The First Loop
1–2 weeks · solo · $0
Minimum lovable version
One complete 3-loop puzzle. Static deterministic clues hardcoded per loop — AI only generates flavor email threads, not the core clues. Loop counter, reset mechanic, and a single satisfying reveal screen on solve.
Leave out
  • ✕Multiple puzzle scenarios in v1 — one well-crafted puzzle is enough
  • ✕AI-generated core clues — only flavor text via AI; clues must be locked
  • ✕Hint system — validate solvability without crutches first
  • ✕Any account system or cross-device state
  • ✕Timer or scoring system
Done when
3 people who did not build it can solve the puzzle without asking you for hints. Not before.
H2 — Scale Later
Loop Archive
After H1 validated · low-moderate budget
3 upgrades
  • +Puzzle anthology — 5 cases. Each with a different workplace setting and a distinct loop mechanic variation (more loops, branching clue paths, one red herring loop).
  • +Tiered hint system. 3 hints per puzzle, gated by loop count. Only add after you know players are stuck — not before you know where they get stuck.
  • +Supabase solve tracking. Which puzzles are solved, average loops used, where people give up. This data drives which cases get built next.
New risks
  • !Each new puzzle requires its own clue graph — non-trivial design time, not just writing.
  • !Players may expect consistent puzzle quality across cases — one weak case damages all of them.
Skills to gain first
Supabase basic tables Puzzle clue graph design Playtest methodology
H3 — Moonshot
Loop Engine Platform
Small team · 12+ months post-H1
Vision
A creator toolkit that lets mystery writers build loop-inbox puzzles without code — a no-code puzzle authoring tool plus a player storefront. Writers publish cases; players subscribe or pay per case.
Roles needed
Puzzle designer / writer Platform engineer Creator community lead
Budget assumption
$20K–$50K. Creator tooling is expensive to build right. Revenue potential is real (subscription tiers) but requires a catalog of quality cases first — at least 10 before a storefront is credible.
Why only later
A creator platform with no proven player audience is a field with no farmers. Prove players exist and return before building tools for writers who serve them.
Trigger: H1 → H2
At least 20 players have solved the first puzzle and at least 5 of them asked "is there another one?" That question, unprompted, is the only valid trigger.
Budget trigger: None — H2 is design time, not cash. People trigger: Solo still.
Trigger: H2 → H3
5-case archive has a measurable return-player rate above 40%, and at least 2 writers have asked how they could submit their own puzzles.
Budget trigger: $10K+ available or a revenue model validated. People trigger: 1 puzzle writer + 1 engineer.
De-scope under timeline pressure
Cut all AI content generation from H1. Fully static hardcoded puzzle. Ship the mechanic, not the content pipeline. Add AI flavor emails only in H2.
Do not start H2 yet if
The first puzzle has not been solved successfully by someone other than you. An unvalidated clue graph is not a foundation — it's a debt.
Market angle
A mystery game where email threads evolve across time loops — no dialogue, no exploration, no NPCs — the mechanic has no named competitor.
Post-market #3 · Final score 4.57 · Game · Tier 1
Inventory Oracle
Game Build Now Edge 5/5
Moonshot 3 of 5: A daily object-deduction puzzle in the Wordle format — one inventory published each day, a single global solve count, designed for adult mystery fans who want a 5-minute mental puzzle with no fluff.
H1 — Build Now
Proof of Deduction
1 week · solo · $0
Minimum lovable version
5 pre-designed inventory puzzles. Each is 6–10 items shown as a clean list. Player submits a free-text answer (what happened / who is this person). AI scores the answer against a locked solution. Single pass — no loops, no hints.
Leave out
  • ✕AI-generated puzzles — hand-craft every puzzle in v1
  • ✕Difficulty tiers or scoring system
  • ✕Reveal animation or polish
  • ✕Any account or session persistence
  • ✕Share results mechanic
Done when
5 hand-crafted puzzles are solvable by someone who did not design them, with AI scoring their answer correctly and fairly. Deduction works or it doesn't — this test answers it.
H2 — Scale Later
Puzzle Archive + Daily Format
After H1 validated · low budget
3 upgrades
  • +Daily puzzle format. One new inventory per day, date-seeded. Share-result card (no spoilers — just "solved in X guesses"). Wordle distribution of the daily format, applied to deduction.
  • +Difficulty calibration tiers. Beginner (obvious item set), intermediate (ambiguous items), expert (misleading items). Tags per puzzle — player chooses on entry.
  • +AI-assisted puzzle generation. Claude generates candidate inventory sets from a category seed (e.g. "1940s traveler" or "failed heist"). You review and approve each — AI drafts, you curate.
New risks
  • !Daily format creates a content treadmill — you need 30+ puzzles banked before launching daily.
  • !AI scoring of free-text answers needs calibration — a harsh scorer kills the fun, a lenient one removes challenge.
Skills to gain first
Date-seeded puzzle delivery Share card generation AI answer scoring prompt design
H3 — Moonshot
Object Oracle Daily
Solo-to-small-team · 12+ months post-H1
Vision
A daily deduction puzzle that reaches the Wordle / NYT Games audience — adults who want a brief, satisfying mental exercise. One puzzle per day. Global leaderboard of correct solves. Optional paid archive access for subscribers.
Roles needed
Puzzle designer (part-time) Backend for daily delivery
Budget assumption
$5K–$15K. Low infrastructure cost — static daily content plus simple Supabase. Main cost is puzzle curation time and potentially a part-time puzzle writer.
Why only later
Daily games succeed or fail on puzzle quality sustained over months. The treadmill is punishing. You need a puzzle pipeline that works before you need a daily audience.
Trigger: H1 → H2
Players are solving the 5 puzzles AND asking for more — specifically requesting a puzzle they can return to daily. That request, unprompted, is the signal to build the daily format.
Budget trigger: None. People trigger: Solo.
Trigger: H2 → H3
The daily format has 100+ daily active solvers for at least 2 consecutive weeks — not launch-day traffic. Sustained daily return is the only valid moonshot trigger for this format.
Budget trigger: Evidence of willingness to pay (survey or a single $2 "tip"). People trigger: 1 puzzle writer contracted.
De-scope under timeline pressure
Cut free-text answers. Use multiple-choice in v1 — 3 options, one correct. Removes AI scoring entirely. Reintroduce free text in H2 once the puzzle quality is proven.
Do not start H2 yet if
You have not validated AI answer scoring works fairly on at least 20 test answers per puzzle. Bad scoring is worse than no scoring — it breaks trust on the first solve.
Market angle
A mystery game with no dialogue, no NPCs, no exploration — you read only objects, and you deduce everything. The mechanic has no direct named competitor.
Post-market #4 · Final score 4.57 · Tool · Tier 3
Mythwright
Tool Build Later Edge 4/5
Moonshot 4 of 5: The first AI-powered worldbuilding platform with a live consistency layer — not a wiki, not a note tool, but an active referee that keeps your world honest as it grows.
H1 — Build Now
Lore Atlas
2 weeks · solo · $0 · Tier 2 version
Minimum lovable version
Entry form with tag system and free-text fields. On each new submission, AI runs a consistency check against existing entries. Flags potential contradictions inline. All stored in localStorage. No graph — flat list view only.
Leave out
  • ✕D3 relationship graph — add only after 20+ entries prove the flat list is insufficient
  • ✕Collaboration or multi-user features
  • ✕Export to game engine formats
  • ✕Semantic search or embeddings
  • ✕Any visual world map
Done when
You or another worldbuilder has added 20+ entries and the AI has caught at least 2 genuine contradictions they didn't notice themselves. That's the product working.
H2 — Scale Later
Mythwright Core
After H1 validated · low-moderate budget
3 upgrades
  • +D3 relationship graph. Visualize connections between lore entries — factions, locations, characters, events. The graph becomes navigable with enough entries. Do not add before 20+ entries exist.
  • +Connection suggestions. After adding an entry, AI suggests which existing entries it likely relates to — not just contradiction detection but active link-building.
  • +Supabase + export. Cross-device, persistent, and downloadable as Markdown. The output format writers actually use.
New risks
  • !D3 graph becomes unreadable past 50 nodes without significant layout engineering.
  • !AI connection suggestions may generate false links that feel authoritative — users may trust them without verification.
Skills to gain first
D3 force graph layout Supabase row-level security Markdown export
H3 — Moonshot
Mythwright Platform
Small team · 12–18 months post-H1
Vision
Multi-user collaborative worldbuilding with an AI referee that tracks consistency across contributors. Each writer sees a live conflict log. Campaign groups, novel co-authors, and game designers pay for team plans.
Roles needed
Full-stack engineer Graph UX designer Community (writers/TTRPG)
Budget assumption
$25K–$60K. Real-time collaboration on a lore graph is technically non-trivial. The audience (TTRPG groups, writing communities) is willing to pay for team tools if trust is established first through the solo version.
Why only later
Collaboration on shared creative documents is a hard product problem — permissions, merge conflicts, trust. Solve it for one person first. The single-user version earns the right to become collaborative.
Trigger: H1 → H2
You or another user has built a lore atlas with 20+ entries, used it for an active project, and found yourself wanting to visualize the relationships — not just list them. That friction is the graph's call to exist.
Budget trigger: None. People trigger: Solo; or one writer using it for their own project.
Trigger: H2 → H3
Multiple writers or a TTRPG group ask whether they can use it together — not whether it could theoretically support collaboration, but that they actually want to right now.
Budget trigger: $10K+ or a paying team willing to fund the feature. People trigger: 1 engineer + 1 design collaborator.
De-scope under timeline pressure
Cut AI connection suggestions — keep only contradiction detection. One AI function is testable; two competing AI functions are confusing. Build the second only after the first is trusted.
Do not start yet if
You have not shipped Prompt Forge or at least one other tool first. Mythwright is Tier 3 difficulty and requires Supabase and D3 fluency you may not have yet. Build skills before building this.
Market angle
An AI consistency referee for worldbuilders — actively flags contradictions as you write, not a passive wiki you manually maintain.
Post-market #5 · Final score 4.57 · Game · Tier 3 · Flagged
Voice Dungeon Master
Game Build Later Market flagged
Moonshot 5 of 5: The first voice-narrated solo TTRPG experience with cross-session memory — a specific fantasy world, a persistent cast, narrated by a distinct AI voice — not a generic AI DM but a named, characterful game product.
H1 — Build Now
The Text DM (No Voice)
2 weeks · solo · low cost · text only
Minimum lovable version
Text-only GM chat loop in a specific locked world (not generic fantasy). One 3-act adventure structure. Session state saved to localStorage — resume next session exactly where you left off. No voice yet. The persistence is the differentiator, not the voice.
Leave out
  • ✕ElevenLabs voice — validate the narrative and persistence first
  • ✕Character sheets, dice UI, or any TTRPG mechanical layer
  • ✕DALL-E scene images
  • ✕Multiple world settings
  • ✕Any account system
Done when
You complete a full 3-act session, save it, close the browser, return the next day, and the GM picks up the narrative without any loss of context. That is the entire v1 promise.
H2 — Scale Later
Voice + World
After H1 validated · moderate budget · ElevenLabs
3 upgrades
  • +ElevenLabs narration. One specific narrator voice — not player-selectable in v2. The GM has a voice. This is the differentiating feature that no text-only AI DM can match.
  • +Session recap card. At session start, the GM reads a 3-sentence recap of the last session aloud. Solves the re-entry problem that kills long narrative games.
  • +Supabase session history. Cross-device persistence. The campaign travels with you — not locked to one browser on one machine.
New risks
  • !ElevenLabs API cost compounds fast at sustained session use — benchmark cost per session before shipping.
  • !Context window fills before a long campaign resolves — need a summarization layer to compress old sessions.
  • !Voice latency may break narrative immersion — acceptable threshold is under 3 seconds per narration block.
Skills to gain first
ElevenLabs streaming API Context summarization prompts Supabase session storage Cost-per-session tracking
H3 — Moonshot
Named World Game Product
Small team · 18+ months post-H1
Vision
A named solo TTRPG product set in one specific world — with a distinct narrator voice, a persistent cast of NPCs, and a campaign structure that evolves with your decisions. Sold as a premium product ($15–$30 one-time or $8/month), not a tool.
Roles needed
Narrative designer Voice director (ElevenLabs tuning) Backend engineer
Budget assumption
$30K–$80K. ElevenLabs at scale is the primary ongoing cost. Revenue must offset API costs — a subscription model is required, not optional. This is a product business, not a solo side project at this horizon.
Why only later — and the flag is real
The flagged market risk is not resolved by H3 ambition — it is resolved by sharp world-setting specificity in H1 and H2. A generic AI DM is invisible. A named world with a distinct narrator and a persistent cast is a product. Do not start H3 without that identity locked.
Trigger: H1 → H2
You have completed at least 3 sessions across different days, confirmed the persistence works flawlessly, and found that the absence of voice is a genuine frustration rather than an acceptable limitation. That specific frustration is the only valid trigger to add ElevenLabs.
Budget trigger: Budget $20–$50/month for ElevenLabs API during testing. People trigger: Solo.
Trigger: H2 → H3
The H2 product has paying users (even at $1) who return weekly, AND you can measure that the specific world-setting is the reason they chose this over other AI narrative tools. Identity confirmation is the trigger — not audience size alone.
Budget trigger: Revenue covering API costs, or $15K+ available. People trigger: 1 narrative designer + 1 engineer.
De-scope under timeline pressure
Cut the 3-act structure. Ship a single open-ended session with no defined end — just persistent chat with GM memory. One act, unbounded, saved. Structure comes in H2.
Do not start yet if
You have not shipped at least one complete project from the Build Now tier. Voice DM is the most technically and creatively complex item in the top 5. It requires skills this portfolio has not yet proven. Start here last, not first.
Market angle
Not a generic AI DM — a specific voiced world with persistent campaign memory. The world name and narrator identity are the product, not the AI underneath it.
