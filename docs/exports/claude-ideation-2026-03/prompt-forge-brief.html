Product vision
A personal prompt testing workbench for solo AI builders. You write a prompt, define a rubric, run it against Claude, rate the output, and save the winners. Over time it becomes your reusable prompt library — the difference between iterating randomly in chat and actually improving systematically.

It does not try to be a team tool, a marketplace, or an analytics dashboard. It is a single-player instrument for one job: knowing which of your prompts actually works and being able to find it again.
Must-have v1 features
  • 1
    Prompt runner.Text input area, send to Claude API, display raw output. The whole tool lives or dies on this working cleanly.
  • 2
    Rubric scorecard.User defines 3–5 criteria before running (e.g. clarity, tone, specificity). After output appears, rate each criterion 1–5. This is what separates it from just using Claude chat.
  • 3
    Save to library.One-click save of the prompt + output + rubric scores + optional notes. Persisted to localStorage. Without this, every session is throwaway.
  • 4
    Library browse + search.List of saved prompts filterable by tag and sortable by score. The library is the compounding value — it must be findable.
  • 5
    Tag system.Free-text tags per prompt (e.g. "game", "narrative", "system"). Minimum viable taxonomy. No categories, no hierarchy — just tags.
Nice-to-have later
  • A
    Side-by-side variant comparison.Run 2–3 variants of a prompt, score all of them, see diffs. Useful after v1 proves the rubric idea has value.
  • B
    Prompt versioning.Track edits to a saved prompt over time. Only worth building after the library has enough entries to make version history meaningful.
  • C
    Export to JSON.Download your full library. Useful for backup or sharing — add after the library is worth exporting.
  • D
    AI-suggested improvements.After rating, ask Claude to suggest a rewrite of the prompt based on the rubric gaps. Meta-layer — do not add before the base loop is stable.
  • E

Market reality check
This is for solo AI builders — developers, indie creators, and content professionals who already use Claude regularly and are frustrated that their prompt iterations live scattered across chat sessions with no way to compare or retrieve them. The audience is small but highly self-aware about the pain: they know they are losing good prompts and making the same mistakes repeatedly. The tool wins by being personal and frictionless rather than team-oriented and feature-heavy — most prompt management tools aim at teams or enterprises (PromptLayer, LangSmith, Helicone) and bring significant setup overhead. The honest uncertainty: the audience may decide that a well-organized Notion doc solves this adequately, and there is no strong evidence yet that they will use a dedicated tool over their existing workflow. The v1 test is whether using the rubric + save loop produces a measurably better prompt within a single session. If it does not, the tool has no argument.
Core user flow
1 · Open the tool
User lands on the runner screen. Sees the prompt input, their rubric (pre-filled from last session or blank), and a library count badge in the corner. No login, no onboarding — immediate.
2 · Define or load rubric criteria
User sets 3–5 criteria before running. These can be free-text (e.g. "stays in voice", "gives a concrete example") or chosen from a preset list. Rubric is persistent across sessions.
3 · Write prompt and run
User types or pastes a prompt. Hits run. The output streams in. The UI does not disable or lock during generation — user can start writing notes immediately.
4 · Score the output against rubric
Once output is complete, the rubric scorecard activates. User rates each criterion 1–5. An overall composite score calculates automatically. User can add a short note.
5 · Save or discard
User saves to library with tags, or discards and iterates. If they save, the entry appears immediately in the library list. If they discard, the runner resets to empty — prompt text is not cleared until they choose to.
6 · Browse library
User switches to library view. Sees saved prompts sorted by score (default) with tag filters and a search box. Each entry is expandable to show full prompt, full output, and rubric breakdown. No pagination in v1.
7 · Reload and reuse
User clicks a saved prompt to load it back into the runner. From there they can edit, re-run with a modified version, or just copy the text. This is the core payoff — the library feeds the runner.
Data / state model — plain English
Everything stored in localStorage as JSON. No server, no auth. These are the four objects the app needs to function.
Prompt Entry — one saved item in the library
  • idauto-generated timestamp string — unique key, used for lookup and deletion
  • promptTextthe full prompt string the user wrote
  • outputTextthe full raw response from Claude
  • rubricScoresarray of {criterionLabel, score} objects — one per rubric criterion
  • compositeScoreaverage of all rubric scores, calculated at save time, stored as a float
  • tagsarray of strings — free text, lowercase, trimmed
  • notesoptional short string the user added — can be empty
  • savedAtISO timestamp — used for sort by recency
Rubric Template — persists across sessions
  • criteriaarray of strings — the user's current active rubric criterion labels
  • updatedAttimestamp — so the app can warn if the user changed the rubric after saving older entries
Session State — in-memory, not persisted
  • currentPromptwhat is currently typed in the runner input
  • currentOutputthe last response from Claude — cleared on new run
  • isRunningboolean — true while Claude API call is in flight
  • currentScorespartial rubric scores entered so far this session — cleared on save or discard
  • activeView'runner' or 'library' — drives which screen is visible
Library Index — persisted, drives search and filter
  • entriesarray of all saved Prompt Entry objects — source of truth
  • allTagsderived set of all unique tags across entries — rebuilt on each save, used for filter UI
  • sortMode'score' | 'recency' — user's last chosen library sort order, persisted
UI screen list
Runner
Primary screen. Prompt input, run button, output display, rubric scorecard (activates post-output), save/discard buttons. Lives at the top of the single-page layout.
Rubric editor
Inline panel on the runner screen — not a separate page. Expandable. User adds, edits, or removes criteria. Criteria persist immediately on change.
Library list
Second primary view. Sorted list of saved entries with composite score, preview snippet, and tag pills. Tag filter row at top. Search box. Sort toggle (score / recency).
Entry detail
Expanded view of a single saved entry — full prompt, full output, rubric breakdown table, notes, tags. Inline on the library list (accordion) or a panel — decide after seeing how long outputs render.
Empty states
Two: empty library (first use prompt) and empty rubric (nudge to add at least one criterion before running). Both are single-sentence nudges, not wizards.
Error states
API error (inline, non-modal), empty prompt submit (field highlight only), localStorage full (rare — warn and suggest export). No full-screen error states.

Not used v1
Supabase Auth Streaming responses Server-side proxy Any third-party service
One security note: calling the Anthropic API directly from the client means the API key is exposed in the browser. For a personal-use-only tool this is acceptable. If you ever share the deployed URL publicly, add a lightweight server-side proxy (a single Vercel Edge Function or Netlify Function) that holds the key and forwards requests. Do not add this in v1 — solve it only when the sharing use case exists.
Test plan
Happy path — what must work for v1 to be shippable
  • ·User opens app fresh — runner screen loads with empty prompt field and rubric editor visible.
  • ·User adds three rubric criteria — they persist after page refresh.
  • ·User types a prompt, clicks run — output appears without error and the run button disables during generation.
  • ·User rates all rubric criteria 1–5 — composite score calculates and displays correctly.
  • ·User adds tags and clicks save — entry appears immediately in library list with correct score and tags.
  • ·User switches to library view — saved entry is visible, tag filter narrows results correctly.
  • ·User clicks saved entry — full prompt and output expand correctly.
  • ·User clicks "load into runner" — prompt text populates the runner input field.
  • ·User refreshes page — library and rubric persist; session state (current output) does not.
Failure path — what the app must handle without breaking
  • ·User submits empty prompt — field highlights red, no API call fires, no error toast.
  • ·API returns error (bad key, rate limit, network) — inline error message appears below output area; app does not crash; run button re-enables.
  • ·User tries to save before rating all criteria — block the save and highlight unrated criteria; do not silently save with 0 scores.
  • ·User adds duplicate tag — silently deduplicate; do not show duplicate pill.
  • ·User deletes an entry — it disappears from the list immediately; if it was the only entry with a given tag, that tag disappears from the filter.
  • ·Library has 50+ entries — search and filter still perform without lag; no pagination required but no jank either.
  • ·User clears localStorage manually — app initializes to empty state without error; no stale references crash the UI.
Edge cases to check but not block launch
  • ·Very long output (4000+ chars) — text wraps correctly, scroll works, expand/collapse does not jank.
  • ·Rubric criteria with long label text — pill wraps or truncates without breaking the layout.
  • ·Mobile-width viewport — runner and library must be usable; not required to be beautiful.
What I might not know yet
Will manual rubric scoring actually feel worth doing, or will it feel like extra work?
This is the core value assumption. If scoring feels like a tax rather than a clarifier, the tool has no argument over a Notion doc.
De-risk first: before building the save system, build only the rubric scorecard in isolation and score 5 real prompts by hand in one sitting. If you find yourself writing more specific prompts to get better scores, the mechanic works.
How fast does the library become useful vs just being a graveyard of old attempts?
Libraries are only valuable if you return to them. If after 20 saved prompts you never load one back into the runner, the whole product is a logging tool, not a workbench.
De-risk: track personally how many times you load a saved prompt into the runner vs writing from scratch after 2 weeks of use. If the ratio is low, the library UX needs work — not more features.
Is localStorage enough, or will it hit size limits with large outputs?
localStorage is capped at ~5MB per origin. A long Claude response is ~2–4KB. You could realistically store 500–1000 entries before hitting the limit, but text can compound fast.
De-risk: add a storage usage indicator in the library header showing bytes used vs the ~5MB cap. If you ever exceed 60%, add output truncation (store first 1000 chars + a "view full" flag) or start building the Supabase path.
What rubric criteria are actually useful vs noise?
If you ship with bad default criteria, every user starts with a broken rubric and has to figure out the right one themselves. That's a cold-start problem that kills habit formation.
De-risk: before coding the rubric editor, manually score 10 of your real prompts using 3–4 different criterion sets. Whichever set produces the most useful signal becomes the default. Ship one good default, not a blank slate.
Will the API key exposure risk become a real problem?
For personal use, calling the API from the client is fine. The key is exposed in network requests but the tool is not shared with strangers. If you post a live demo link publicly, anyone who inspects the requests can extract the key.
De-risk: decide upfront whether this ships as personal-only or as a public demo. If public, add a Vercel Edge Function as a proxy before launch — it's one file and one hour of work. Do not skip this if sharing publicly.

Assumption checklist — validate before coding
  • You have an active Anthropic API key with available credits and have confirmed it works in a basic fetch call.
  • You have personally scored at least 5 real prompts with a rubric by hand — before building — and found the scores useful rather than arbitrary.
  • You know what your first 3–4 rubric criteria will be. They are written down, not vague.
  • You have decided whether this is personal-only or a public demo — and if public, you know you need to add a proxy before sharing the URL.
  • Your Vite + React setup is already working locally and you can run a dev server today without setup friction.
  • You have confirmed localStorage works in your target browser and is not blocked by any extension or privacy setting.
  • You accept that the v1 scope stops at 5 features and are willing to close the "that would be useful" impulse until v1 ships.
Kill-switch criteria
Archive this project quickly if any of the following are true after 2 weeks of real use:
  • You have saved 20+ prompts and never loaded a single one back into the runner. The library is a graveyard, not a workbench — the core loop is broken.
  • Scoring prompts against the rubric takes more than 90 seconds and consistently feels like overhead rather than insight. The friction exceeds the value.
  • You find yourself opening a Notion doc or a plain text file instead of the tool when you want to track a good prompt. The tool has lost the job.
  • The rubric criteria you defined produce identical scores across prompts with meaningfully different quality. The scoring system has no signal — it is theater.
  • After two weeks, the tool has not changed a single prompt you would have written without it. Zero behavior change = zero value delivered.
